## Issue #2 — Graph state-adjoint via exact truncated-exponential transpose (CONFIRM)

The implemented Magnus K=4 map is the **degree-4 truncated Taylor polynomial** `S(τ)=Σ_{m=0..4}Ω₄^m/m!`, not `exp(Ω₄)`. Its transpose factors term-by-term exactly: `S(τ)ᵀ=Σ_{m=0..4}(Ω₄ᵀ)^m/m!`, and since every node operator `A_i=−L_G` (and `L_a=√a L √a`) is symmetric, `Ω₄ᵀ` equals `Ω₄` with ONLY the `√3τ²/12` commutator coefficient sign-flipped (`[A₂,A₁]ᵀ=−[A₂,A₁]`; math.md §42, Theorem 42.1). The resulting state-adjoint is exact, order-4, chart-free, and costs the same as the forward step. It is exposed as a SEPARATE primitive `apply_state_adjoint_into` / `evolve_state_adjoint` on `MagnusGraphHeatChernoff` and `VarCoefMagnusGraphHeatChernoff`, plus a graph-capable `Adjoint(kernel="magnus_graph"|"varcoef_magnus_graph")` Python path. It MUST NOT touch `ChernoffFunction::apply_adjoint_into` (which by contract means genuine `exp(τAᵀ)` and is correctly omitted for the antisymmetric genuine-exp commutator, ADR-0114) and MUST NOT make these types implement `AdjointApply<F>`. Gate: sympy oracle `T_MAGNUS_TRANSPOSE` + a dual-pairing test `⟨Su,g⟩=⟨u,S⋆g⟩`. Risk: LOW (additive methods; gitnexus upstream impact = 0).

## Issue #1 — Adjoint-state parameter-sensitivity primitive `∂(S(τ)u)/∂θ` (CONFIRM the math; REJECT the ML wrapper)

The sensitivity of a terminal functional w.r.t. edge weights `w_{ij}` or timescale `a(·)` is computed by the classical adjoint-state method (Plessix 2006): one forward sweep + one backward costate sweep `λ_k=S_k⋆ λ_{k+1}` using the §42 state-adjoint, with each `∂L_G/∂w_{ij}` a single sparse rank-1 term `(e_i−e_j)(e_i−e_j)ᵀ` (math.md §43). Because the truncated transpose is exact (Issue #2), the discrete adjoint of Magnus-K4 reproduces the solver gradient to machine precision, order-4, with no Lie-group chart dependence. We implement option (a): a first-class MATH primitive returning the forward-mode JVP `∂(Su)/∂θ`, the reverse building block `S⋆(τ)·g`, and the assembled `∂J/∂θ` adjoint-state sum; we ALSO expose option (b)'s read-only generator accessors via Issue #5. **In-core-math vs revssm-ML boundary (NORMATIVE):** IN-CORE = the adjoint-state recursion, rank-1 generator derivatives, JVP/VJP linear-algebra primitives (these are direct consequences of the Chernoff/Magnus derivation). OUT-OF-CORE, stays in `revssm` = `torch.autograd.Function` integration, neural-ODE training hooks, optimizer steps, dtype-for-deployment policy, any framework tape. Litmus test: "is this a mathematical consequence of the Chernoff/Remizov derivation?" → in-core; "is this ML-framework plumbing?" → revssm. Gate: finite-difference match `O(ε²)` + per-step JVP order ≥ 3.9 (`T_ADJOINT_STATE_SENSITIVITY`). Risk: LOW.

**Amendment 1 (math.md §43.2 √a-node correction).** The `T_ADJOINT_STATE_SENSITIVITY` pre-flight oracle caught a mis-stated closed form for the timescale building block BEFORE any Rust was written: §43.2 originally gave `∂L_a/∂a_k = ½ a_k^{-1/2}(E_k L_a + L_a E_k)`, which double-counts one factor of `D = diag(√a)` and holds only at `a ≡ 1`. The correct, now-NORMATIVE form is `∂L_a/∂a_k = ½ a_k^{-1/2}(E_k L D + D L E_k)` (bare combinatorial `L = L_G`, single `D`), equivalently `½ a_k^{-1/2}(E_k D^{-1} L_a + L_a D^{-1} E_k)`; this matches the implemented `L_a = √a L_G √a` (`graph_var_coef.rs` / `varcoef_magnus_graph.rs`). Issue #1's `a`-timescale `apply_param_deriv` MUST use the corrected form. The fix touched only the spec; the oracle's independent symbolic `diff(L_a, a_k)` is unchanged.

## Issue #3 — f32 binding path; fp16 REJECTED (CONFIRM f32, REJECT fp16)

`semiflow-core` is already generic over `SemiflowFloat` (impl'd for f32 and f64 only; `float.rs:73-74`). Bindings expose f64 only. We add an opt-in `dtype="f32"` switch on the graph/diffusion `semiflow-py` kernels, dispatched by a small concrete enum (mirroring `KernelVariant` in `adjoint.rs`; PyO3 cannot be generic over the wheel boundary), returning `numpy.ndarray[float32]`. Default stays `"f64"` (zero behaviour change). FFI/WASM feasibility: same generic core supports it, but the v0.10/v0.11 ABI froze `f64`-only `extern "C"` / JS surfaces (ADR-0028); adding f32 there is a NEW additive ABI surface deferred to a future minor, not in this wave. **fp16 REJECTED (NORMATIVE):** no `impl SemiflowFloat for f16` will be added — (1) `semiflow-core` is at its hard dep budget 3/3 (`num-traits`, `libm`, `num-complex`; constitution Override #1 keeps the ≤3 cap ENFORCED), and any half-precision crate (`half`) is a 4th dep that violates the budget; (2) f16 has insufficient mantissa for the Chernoff convergence regimes the library certifies; (3) half-precision is an ML-deployment concern, not consequence-math (same boundary as Issue #1). Risk: LOW (additive Python surface; no core change).

## Issue #5 — Read-only Laplacian introspection (CONFIRM)

`semiflow.Laplacian` (`PyLaplacian`, `graph_extra.rs`) wraps `Arc<Laplacian<f64>>` whose core type already exposes `row_ptr()`/`col_idx()`/`vals()`/`n_nodes()`. We add read-only `to_dense()` (returns row-major `numpy.ndarray[float64]` shape `(n,n)`) and CSR accessors `row_ptr()`/`col_idx()`/`vals()` (returning copies). All are pure reads returning copies — the frozen-topology invariant (CSR `row_ptr`/`col_idx` immutable, ADR-0048) is PRESERVED because no method exposes a mutable handle. This surfaces the discretized generator as a math object and unblocks Issue #1 option (b). `graph_extra.rs` is at 459 LoC; the four accessors push it toward/over the 500-LoC default cap, so the engineer SHALL extract them into a new `crates/semiflow-py/src/laplacian_introspect.rs` rather than request a new constitution carve-out. Risk: LOW (read-only).
